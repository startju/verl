## 概述

本 commit 引入 partial rollout 实验性功能，新增以下模块：

- `verl/experimental/partial_rollout/agent_loop/agent_loop.py`（PRv3AgentLoopWorker / PRv3AgentLoopManager）
- `verl/experimental/partial_rollout/agent_loop/single_turn_agent_loop.py`
- `verl/experimental/partial_rollout/prompt_manager.py`（RolloutPromptManager）
- `verl/experimental/partial_rollout/ray_trainer.py`（PRv3RayPPOTrainer）
- `verl/experimental/partial_rollout/vllm_rollout/vllm_async_server.py`（PRv3vLLMHttpServer）
- 修改 `verl/experimental/agent_loop/tool_agent_loop.py`（透传 stop_reason）
- 新增 `test.sh`、recipe 子模块指针变更

整体设计思路：通过 `RolloutPromptManager` 维护一个 prompt 队列，agent loop worker 不断 pull/push，配合 `cancel`/`resume` 实现"部分 rollout 可中断 & 恢复"。

**结论：本 commit 不能合入主干。** 存在多处编译/运行期就会立即崩溃的硬伤，强烈怀疑这版代码从未端到端跑通。下面按严重级别列出问题。

## 🔴 P0 — 阻塞性 Bug（必然崩）

### 1. `agent_loop.py:49,67` — `asyncio.create_task` 误用生成器

```python
running_set: set[asyncio.Task] = {
    asyncio.create_task(self._generate_sequences_for_prompt(rp) for rp in rollout_prompts)
}
```

`(... for rp in ...)` 是 generator expression，传给 `create_task` 会立即 `TypeError: a coroutine was expected, got <generator>`。第 67 行同样的写法重复一次。

正确写法：

```python
running_set = {asyncio.create_task(self._generate_sequences_for_prompt(rp)) for rp in rollout_prompts}
```

### 2. `agent_loop.py:54` — `asyncio.wait` 漏 `await`

```python
done, _ = asyncio.wait(running_set, return_when=asyncio.FIRST_COMPLETED)
```

`asyncio.wait` 是协程，必须 `await`。当前代码会 `RuntimeWarning: coroutine '...' was never awaited`，且 `done` 是协程对象，后续 `for task in done` 直接报错。

### 3. `agent_loop.py:58` — 调用了不存在的方法

```python
self.prompt_manager_handle.push_prompt.remote(rollout_prompt)
```

`RolloutPromptManager` 实际方法名是 `push_prompts`（接受 list）。运行时会 `AttributeError`。

### 4. `agent_loop.py:233` — `super()` 调用多传 `self`

```python
return await super().generate_sequences(self, prompts)
```

`super()` 已经绑定 `self`，再传一遍会变成两个位置参数。改为 `super().generate_sequences(prompts)`。

### 5. `agent_loop.py:243-247` — 同步忙等 + `pull_batch` 是 actor 方法

```python
while True:
    output = self.rollout_prompt_manager.pull_batch()
    if output:
        break
```

两个问题：
1. `rollout_prompt_manager` 是 Ray Actor handle，`pull_batch()` 没有 `.remote()`，**这个调用根本拿不到结果**（返回 actor method handle，恒为 truthy，循环只跑一次返回错误对象）。
2. 即使加上 `.remote()` + `ray.get`，也是 100% busy loop，应该用 `await asyncio.sleep(...)` 或事件通知。

### 6. `ray_trainer.py:78-87` — `while need_more` 永远进不去

```python
need_more = False
while need_more:
    try:
        batch_dict = next(data_loader_iter)
    ...
```

`need_more` 初始化为 `False`，循环体永远不执行，导致 `gen_batch` 在第 94 行未定义就被使用 → `UnboundLocalError`。意图显然应该是 do-while；建议：

```python
need_more = True
while need_more:
    ...
```

### 7. `ray_trainer.py:92` — 调用不存在的 `push_pending_prompts`

```python
need_more = ray.get(self.rollout_prompt_manager.push_pending_prompts.remote(batch, gen_batch))
```

`RolloutPromptManager` 没有 `push_pending_prompts`，只有 `push_batch`。`AttributeError` 必触发。

### 8. `ray_trainer.py:72` — async 方法未 await/未 `.remote()`

```python
self.async_rollout_manager.init_agent_loop_workers(self.rollout_prompt_manager)
```

`init_agent_loop_workers` 是 `async def`，这里既没 `await` 也没 `ray.get(... .remote(...))`，调用返回 coroutine 对象后立刻被 GC，方法体根本没执行 → 后续 `agent_loop_workers` 为空。

### 9. `vllm_async_server.py:43-82` — `PRv3vLLMHttpServer.generate` 无限递归

```python
async def generate(self, ...):
    ...
    async def _generate():
        self.token_output_dict[request_id] = await self.generate(  # ← 自己调自己
            prompt_ids, sampling_params, request_id, image_data, video_data
        )
```

调用的是 `self.generate`（即当前方法），不是 `super().generate`。一旦不被 cancel，会无限递归直至栈溢出。

另外 `asyncio.create_task(_generate)`（第 62 行）传的是函数对象不是 coroutine，正确写法是 `asyncio.create_task(_generate())`。

### 10. `vllm_async_server.py:81` — `dict.update` 误用

```python
token_output.extra_fields.update("stop_reason", token_output.stop_reason)
```

`dict.update` 不接受 `(key, value)` 形式的两个位置参数，会 `TypeError`。应为：

```python
token_output.extra_fields["stop_reason"] = token_output.stop_reason
```

### 11. `vllm_async_server.py:51-54` — 返回值类型不一致

```python
async def generate(...) -> TokenOutput:
    async with self.lock:
        if self.paused:
            return None, True
```

签名声明返回 `TokenOutput`，实际返回 `(None, True)` / `(TokenOutput, bool)` tuple。调用方 `single_turn_agent_loop` 把它当 `TokenOutput` 用（`output.token_ids`），运行时会出错。

### 12. `prompt_manager.py:60` — `AgentLoopOutput` 参数缺失 + 共享引用

```python
agent_loop_output_list=[AgentLoopOutput(extra_fields={"stop_reason": "aborted"})]
* self.config.actor_rollout_ref.rollout.n,
```

两个问题：
1. `AgentLoopOutput` 是 Pydantic BaseModel，必填字段有 `prompt_ids`、`response_ids`、`response_mask`、`metrics`。只传 `extra_fields` 会 `ValidationError`。
2. `[obj] * n` 复制的是引用，n 个 list 元素指向同一对象。后续单条 prompt 的 i-th rollout 修改 `extra_fields` 会污染所有 rollout。必须改为 `[AgentLoopOutput(...) for _ in range(n)]`。

### 13. `single_turn_agent_loop.py:44-45` — 续跑分支 `images`/`videos` 未定义

```python

        return last_agent_loop_output

# 3. generate sequences

        image_data=images,   # ← 续跑分支根本没赋值

    )
```

`images`/`videos` 只在首次生成分支里赋值，续跑分支会 `UnboundLocalError`。即使 `multi_modal_data` 在续跑分支里直接用了 `last_agent_loop_output.multi_modal_data`，`images`/`videos` 也得相应 `last_agent_loop_output.multi_modal_data.get("images")` 之类。

### 14. `single_turn_agent_loop.py:82,89` — 访问不存在的属性

```python
if response_logprobs is not None or output.response_logprobs is not None:
    response_logprobs = (response_logprobs or []) + (output.response_logprobs or [])
```

`output` 是 `TokenOutput`，字段名是 `log_probs` 不是 `response_logprobs`。`AttributeError`。

### 15. `prompt_manager.py:91-95` — `is_prompt_done` KeyError

```python
def is_prompt_done(prompt: RolloutPrompt) -> bool:
    for agent_loop_output in prompt.agent_loop_output_list:
        if agent_loop_output.extra_fields["stop_reason"] in ("abort", "aborted"):
```

`extra_fields` 在很多正常完成路径上不会被设置 `stop_reason`（`vllm_async_server.py:81` 的设置语句本身就是坏的，且 `tool_agent_loop.py` 在正常 stop 时把 `stop_reason` pop 掉）。这里直接 `[]` 访问会 KeyError；应该 `.get("stop_reason")`。

### 16. `tool_agent_loop.py:222-223` — pop 未传默认值

```python
if output.stop_reason is not None:
    agent_data.extra_fields["stop_reason"] = output.stop_reason
else:
    agent_data.extra_fields.pop("stop_reason")
```

第一次 `output.stop_reason is None` 时，`extra_fields` 里还没有 `stop_reason` key，直接 `pop` 会 KeyError。应为 `pop("stop_reason", None)`。

## 🟠 P1 — 严重逻辑缺陷

### 17. `agent_loop.py:52-68` — `is_canceled` 一旦置 True 永远不复位

```python
is_canceled = False
while running_set:
    done, _ = asyncio.wait(...)
    for task in done:
        ...
        if not is_prompt_done(rollout_prompt):
            is_canceled = True
    if is_canceled:
        continue
    new_rollout_prompts = ray.get(...)
```

只要任意一次有 prompt 没完成，`is_canceled` 永远为 `True`，从此不再补充新 prompt，最终 `running_set` 抽干即退出。设计意图大概是"cancel 期间不拉新任务，resume 后继续"，但当前代码没有"resume 后清零"的逻辑。

### 18. `agent_loop.py:65` — `len(done)` 复用循环变量

```python
for task in done:
    running_set.remove(task)
    ...
new_rollout_prompts = ray.get(self.prompt_manager_handle.pull_prompts.remote(len(done)))
```

`done` 是 for 循环遍历的 set，`len(done)` 在循环结束后仍是这次 wait 的所有完成任务数（语义其实是对的），但读者很容易误以为是 0。建议显式存到一个 `num_done` 变量。

### 19. `agent_loop.py:148-152` — 同步阻塞调用

```python
rollout_promput.agent_loop_output_list = await asyncio.gather(*tasks)
if is_prompt_done(rollout_promput):
    rollout_promput.gen_batch_output = self._postprocess(rollout_promput.agent_loop_output_list)
    rollout_promput.agent_loop_output_list = []
return rollout_promput
```

变量名打错：`rollout_promput` → `rollout_prompt`（共出现 8 次）。

### 20. `prompt_manager.py:88` — 取消任务回插 pending 队列头部

```python
def push_prompts(self, prompts: list[RolloutPrompt]):
    for prompt in prompts:
        if is_prompt_done(prompt):
            self.done_queue.append(prompt)
        else:
            self.pending_queue.appendleft(prompt)  # ← LIFO
```

被 cancel 的 prompt 立刻插队到 pending 队列头部，会被下一次 `pull_prompts` 立刻取出，可能反复 cancel/resume 同一批 prompt 而其他任务饥饿。建议 `append`（FIFO）或单独的 priority 通道。

### 21. `prompt_manager.py:64` — `need_more` 语义反了

```python
def push_batch(self, batch, gen_batch):
    ...
    return len(self.pending_queue) < 2 * self.batch_size
```

返回 True 表示"队列还少，需要更多"。在 trainer 里：

```python
need_more = ray.get(self.rollout_prompt_manager.push_pending_prompts.remote(batch, gen_batch))
```

语义是"是否还需要继续 push"。OK 看起来一致，但跟前面 P0 #6 的 `need_more = False` 配合下永远不会工作。

### 22. `ray_trainer.py:99` — `gen_batch_output` 与 prompt_manager 流向冲突

```python
gen_batch_output = self.async_rollout_manager.generate_sequences(gen_batch_output)
```

`PRv3AgentLoopManager.generate_sequences` 内部完全忽略 `prompts` 参数（除了 `meta_info["validate"]` 标志），从 `rollout_prompt_manager` 直接 `pull_batch`。这里把 `gen_batch_output` 传进去其实是误导，且 `gen_batch_output` 自身的 `meta_info`（`global_steps`、`validate`）没有正确传到 prompt manager 内部的 prompt。建议明确两个流向（验证走原路径，训练走 prompt manager），或者完全去掉这个参数。

### 23. `agent_loop.py:251-254` — `metrics` 提取逻辑可疑

```python
outputs = [output]
metrics = [output.meta_info.pop("metrics") for output in outputs]
timing = self._performance_metrics(metrics, output)
output.meta_info = {"timing": timing, **outputs[0].meta_info}
```

只有一个 output 时，`outputs = [output]` 是冗余包装。`pop` 已经从 `output.meta_info` 移除了 metrics，最后行的 `**outputs[0].meta_info`（与 `output.meta_info` 同对象）确实不再含 metrics，这是想要的——但读起来很绕。简化：

```python
metrics = output.meta_info.pop("metrics")
timing = self._performance_metrics([metrics], output)
output.meta_info["timing"] = timing
```
